\section{Introduction}
Public key cryptography is a ubiquitous component of modern security.
However, it relies on the assumption that private keys can only be used by their intended owner, which is difficult to guarantee in reality.
When this assumption is true, signatures generated by private keys have trustworthy origin,
and messages decrypted by private keys have trustworthy readers.
In small-scale, personal settings, users may reasonably trust one another to protect their keys.
In contrast, large-scale systems such as web servers handle thousands of requests per second, 
and keys must be both highly available and strongly protected.
Thus, the security of their keys depends entirely on machine configuration and automated processes.
A root-level compromise to such systems is catastrophic for the security of keys.
This raises a critical question: how can we protect a secret, e.g. a private key, on a general-purpose system from a root-level attacker such that only the owner of the secret can use it?

We present a software-level framework around the Trusted Platform Module (TPM) 2.0 hardware security module \cite{Trusted-Computing-Group:2025:TPM-2.0-Library} to answer this question.
The TPM provides a root of trust of storage for secrets and cryptographic keys,
and can perform cryptographic operations outside of main system memory so keys are never directly exposed.
The TPM supports authorization checks to access these secrets and keys.
However, it notably lacks integration with OS concepts such as users or programs
and has no built-in mechanism that meaningfully \emph{authenticates} such OS entities to restrict key usage.
This lack of integration is the key problem our framework addresses.

At the same time, the TPM supports measured boot which establishes firmware and software integrity.
Thus, we can measure our software-level authentication framework to ensure it is being enforced before allowing keys to be used.
Our authentication framework uses mechanisms that are built into the Linux kernel: Security-Enhanced Linux (SELinux) \cite{Loscocco:2001:SELinux},
the Linux Integrity Measurement Architecture (IMA) \cite{Sailer:2004:Integrity-Measurement-Architecture}
the Extended Verification Module (EVM) \cite{Zohar:EVM:2010} and Linux Unified Key Setup (LUKS) \cite{Fruhwirth:LUKS:2005} encryption.
Other than using the commonly-available TPM for key storage and measured boot,
we use no specialized hardware nor custom kernel modifications.
We provide background on these mechanisms in \zcref[S]{sec:background}.

In this work, we develop a strategy against a root-level adversary (\zcref[S]{sec:adversary-strategy})
and demonstrate how to combine these existing Linux features---SELinux, IMA, EVM, and LUKS---with
measured boot and TPM policy authorization to enforce fine-grained access control to TPM-resident secrets
to implement that strategy (\zcref[S]{sec:implementation}) on Fedora Linux \cite{FedoraLinux}. 
We demonstrate our framework's successful protection of secrets when the Linux kernel is uncorrupted (\zcref[S]{sec:evaluation}).
We discuss limitations (\zcref[S]{sec:limitations}) of our framework, primarily related to the effectiveness a deep kernel corruption or attacks capable of leaking arbitrary memory.
We finally describe variations (\zcref[S]{sec:variations}) of our setup, 
including modifications that could reduce the complexity of our framework,
which leads to our recommendation that deeper kernel support for underutilized TPM features could enable stronger designs.

\section{Adversary and Strategy}\label{sec:adversary-strategy}
Our motivating example is producing cryptographic signatures,
where the important secret is a private key.
We are concerned about an adversary using that private key to sign falsified data.
We want to ensure only the intended owner of the private key can actually use it,
justifying the trustworthiness of any generated signatures.
In our case, the entity with ownership of the key is a program named \tpmsign.
In general, our strategy can be applied to other kinds of secrets and is not limited to keys.

We assume an adversary has root privileges on a general purpose Linux system.
Without our framework, the adversary's capabilities include reading or writing any file,
installing kernel modules, dumping program memory, modifying the boot setup, and rebooting the system;
thus, they can access the TPM and any files necessary to make a particular request, including using our key.
We assume the adversary has no physical access to the machine, so the hardware and firmware are trustworthy.

To make our implementation generally applicable, we have the following goals:
\begin{itemize}
\item We want to avoid artificially restricting access to the TPM or other system components.
\item We should minimize our framework's impact on a legitimate user or administrator as much as possible.
\item We do not want to assume it is acceptable to lock down the OS such that the owner of the system has little to no agency.
In other words, we do not want to outright disable or extensively limit root access.
\end{itemize}

Due to the limitations of the TPM our framework addresses, our strategy takes the following form:
If a secondary secret is required to use our key and that secret is well-protected,
it authenticates entities by distinguishing entities that can (i.e.\ \tpmsign) and cannot access the secret.
If those entities have integrity, then only the intended, legitimate entities can access the secret to use the key.
This suggests the secret must only be available (LUKS) when access controls (SELinux) and integrity (IMA, EVM) are being enforced (measured boot).
Importantly, these protections must apply to root-level users,
and must prevent alternative means of leaking the secret like installing kernel modules or inspecting program memory.
This strategy ensures the OS authenticates entities accessing the key, working around the lacking OS--TPM integration.
Any of our chosen mechanisms (including the TPM) could be swapped for an alternative design with the same principles.

\subsection{Trusted Platform Module (TPM) 2.0}
The TPM 2.0 is widely available on modern x86 systems,
either as firmware TPMs (fTPMs) integrated in AMD and Intel platforms or as discrete hardware modules,
making it a practical target for our work.
The behavior of the TPM is standardized by the Trusted Computing Group (TCG) \cite{Trusted-Computing-Group:2025:TPM-2.0-Library}.
The TPM serves as a root of trust of storage and root of trust of reporting.
When used with Secure Boot-enabled firmware, it is used to store boot measurements.

\subsubsection{Recording Boot Measurements}
The TPM utilizes Platform Configuration Registers (PCRs) to store measurements \cite{Trusted-Computing-Group:2025:TCG-TPM-2.0-Part1-Architecture}.
PCRs store hashes of values through hash extension operations.
If a PCR has a current hash value $x$ and it is being extended with a new hash $y$,
the PCR's value becomes $x_{new} = \hash{\mathop{concat}(x,y)}$.
By correlating each hash extension to its source,
successfully reconstructing a PCR value demonstrates integrity of the recorded measurements.
For cryptographically secure hash functions like SHA-256,
finding $y$ such that $x_{new}$ lies about previous measurements is infeasible.

PCR extension is the primary mechanism supporting Secure Boot and other measured/trusted boot techniques.
During boot, the UEFI measures the firmware and the bootloader,
and the bootloader measures the kernel and parts of the OS.
Because users cannot interfere with early boot,
the measurements reliably indicate firmware and software integrity,
and they remain trustworthy at runtime as they are stored in PCRs 0--15, which are non-resettable and persist until reboot \cite{Trusted-Computing-Group:2023:TCG-PC-Client-PFP}.

\subsubsection{Loading and Storing Secrets}
Since the TPM has limited storage, secrets are often unloaded and reloaded,
and they are stored as a hierarchy \cite{Arthur:2015:A-Practical-Guide-to-TPM-2.0-Certifying-Keys,Trusted-Computing-Group:2025:TCG-TPM-2.0-Part1-Architecture}.
Primary keys are deterministically generated and act as root parents in the hierarchy.
Their children are regular, randomly generated keys and other secrets.
Parent keys encrypt their children for external storage,
so they must be regenerated (primary keys) or reloaded (regular keys) to decrypt their children.

To improve performance of reloading keys, the TPM can reload a session where the key was already loaded if the system has not rebooted.
Keys can also be persistently stored in the TPM's non-volatile memory (NVRAM) for a further minor improvement.
Loading a key in the TPM merely caches it; authorization checks are still enforced upon use.

\subsubsection{Protecting Keys and Other Secrets}\label{sec:key-protections}
TPM-resident private keys are only usable within the TPM itself and are never available on the OS.
Other TPM-resident secrets have the same protections available, but get released to the OS when access requirements are satisfied.
We focus on TPM-resident keys, without loss of generality.

TPM-resident keys support conditional usage,
optionally requiring a password, HMAC, particular PCR values,
or a more general \emph{policy} to be satisfied.
Password- or HMAC-based authorization requires a secret to be known at use time,
but such secrets have no inherent protection against disclosure.
PCRs effectively limit key usage to a pre-approved boot configuration.
Importantly, keys are tied to their authorization mechanism.
For password-authorized keys, it is possible to change the password.
However, for simple PCR-based authorization,
the approved values cannot be changed,
which is problematic for system updates \cite{Arthur:2015:A-Practical-Guide-to-TPM-2.0-Certifying-Keys,Trusted-Computing-Group:2025:TCG-TPM-2.0-Part1-Architecture}.

Policy-based authorization is more advanced.
Policies can include password and PCR authorization and can also check additional system details like time or boot count.
Flexible policies can be made via ``wildcard'' policies \cite{Arthur:2015:A-Practical-Guide-to-TPM-2.0-Certifying-Keys}.
Wildcard policies must be signed by an administrator with a private key,
and the corresponding public key is included in the TPM-resident key's creation details.
Thus, wildcard policies provide a way to change PCR requirements
so keys remain usable even when boot measurements change.
Wildcard policies may also be leveraged to allow multiple policies to authorize usage of a single key,
or to utilize the same policy for different keys, including keys on different devices.

Notice we did not point out a mechanism for the TPM to directly authenticate OS entities.
TPM policy does not have any straightforward implementation for this.
In practice, any entity allowed to access the TPM (e.g.\ the root adversary) can make requests in place of other entities without distinction, unless manual user input such as a password is required,
and any currently viable automated option relies on the OS to perform authentication completely independently of the TPM.

Two potential improvements are feasible.
The TPM has a feature called ``localities'' which is intended to be used to indicate the origin of a command (hardware, kernel, user-level software) \cite{Trusted-Computing-Group:2023:TCG-PC-Client-PFP,Arthur:2015:A-Practical-Guide-to-TPM-2.0-Certifying-Keys}.
In theory, localities could be tied to entities to restrict access to a key.
However, the TPM trusts the OS to control localities, and there is no existing mechanism, SELinux included, designed to accomplish this.
Furthermore, while technically feasible, (ab)using localities in this way seems questionable according to the standards set in current specification documents.
Alternatively, a resettable PCR could be controlled by the OS to indicate the entity currently accessing the key.
There is also no mechanism to accomplish this currently implemented in Linux, but this would be valid according to specifications.
We designate these proposals as future work since they involve bespoke kernel-level modifications, and discuss the benefits to our framework in \zcref[S]{sec:variations}.

\subsection{Security-Enhanced Linux (SELinux)}
SELinux \cite{Loscocco:2001:SELinux} provides mandatory access control on Linux.
Mandatory access control restricts permissions via rules in a system-wide policy.
In the typically more familiar \emph{discretionary} access control, permissions are set by the owner of the resource.
For example, discretionary access control aligns with POSIX-style file permissions.
While these two concepts are not mutually exclusive, they tend to have distinct implementations in practice.

Labels added to users, files, and processes determine their permissions according to the SELinux policy installed on the system.
These permissions can restrict actions such as reading, writing, and appending certain files,
executing programs, performing inter-process communi\-cation, and opening network ports.
This can override the standard permissions mechanism on Linux.
For example, SELinux can prevent a user's program from accessing an unrelated file owned by the same user,
which would be allowed by the standard permissions mechanism.

\subsection{Integrity Measurement Architecture (IMA)}
IMA \cite{Sailer:2004:Integrity-Measurement-Architecture,IBM:IMA-Documentation} measures and appraises files,
preventing them from being opened if they are changed from an approved state.
Integrity hashes and signatures for files are stored as extended filesystem attributes.
When enabled in \fix mode, IMA updates hashes of files matched by IMA policy.
In \appraise mode, IMA prevents opening appraised files when their contents do not match the associated hash.
Files can also be measured without necessarily being appraised; measurements are kept in a log alongside a correlated PCR.
IMA \fix mode can only be enabled when Secure Boot is disabled in the UEFI,
otherwise \appraise mode is enabled.
IMA policy can include various specifications such as the file owner, the active user, and whether files are being opened for reading or execution.

Signatures can also be used to detect file corruption.
Allowing hashes to be overwritten in \appraise mode is unsafe, as an adversary may know the expected value.
To switch to \fix mode from \appraise mode to update hashes, a reboot is required.
In contrast, signatures can be updated safely in \appraise mode because the administrator's signing key remains protected.
This makes it easier to install and update programs and files.

The first custom IMA policy that is loaded overwrites any default policies.
Afterwards, the custom IMA policy cannot be overwritten, only appended to.
When a file is being checked by IMA, the first matching rule in the policy is applied.
Thus, loading any more policies can only increase strictness of the running policy.

\subsection{Extended Verification Module (EVM)}
\label{sec:evm}
EVM \cite{Zohar:EVM:2010,IBM:IMA-Documentation} protects file meta\-data, such as the permissions and SELinux label.
EVM can use HMACs to detect any offline filesystem modifications, or use signatures to detect runtime modifications;
both can be enabled at once.
When a file is appraised by IMA, its EVM HMAC/signature is verified.

``Portable'' EVM signatures exclude system-specific metadata and actively prevent modifying file metadata.
Thus, adversaries cannot edit file metadata to bypass protections.
This differs from the behavior of IMA, which checks if a file is corrupt upon access,
but does not inherently prevent the corruption from occuring.
When both HMACs and signatures are enabled, EVM may replace non-portable signatures with HMACs.

\subsection{Linux Unified Key Setup (LUKS)}
LUKS \cite{Fruhwirth:LUKS:2005} creates encrypted filesystem images, protected with a password or TPM PCRs.
Thus, LUKS can protect files when the system boots into an unapproved state.
Whether a LUKS-based or a manual decryption process is used, a key must be kept in memory or otherwise easily available.
Since LUKS containers trivially retain file metadata and are managed by the kernel,
we favor the widespread, easy-to-use LUKS mechanisms over a manual decryption process.

\section{Framework}\label{sec:implementation}
In our framework, access to a TPM signing key \pk is restricted to a single program \tpmsign.
SELinux enforces rules on \tpmsign and a wildcard policy \wcp.
IMA protects the integrity of \tpmsign, as well as any binaries permitted to launch it.
EVM and SELinux protect the SELinux labels on \tpmsign and \wcp.

The wildcard policy \wcp is the secret used to access \pk.
\wcp is stored within a LUKS container that unlocks during boot when our framework is enabled.
When the system is booted into an unapproved state, \wcp is sealed in the LUKS container and cannot be accessed.
Additionally, \wcp itself also requires the PCRs to indicate the system booted with our framework.
If the container is unlocked manually while \wcp is unprotected, the PCRs will indicate such, so \pk remains unusable.
Thus, for an adversary to access \wcp, they must exploit the running system to bypass the IMA or SELinux policies or leak \wcp from memory.

\emph{Deny rules} are available in the SELinux Common Intermediate Language (CIL) policy format.
They provide a clear and robust way override existing policy to restrict permissions,
and prevent accidental allowances in future policies.
Sometimes assignments like \verb|files_type(x)| which we used above have some default behavior which is too permissive.
Deny rules allow us to fix this easily.
We use deny rules to prevent any entity from relabeling malicious programs to \exect, reading or relabeling \filet files,
or tracing \domaint processes. We prevent \verb|LD_PRELOAD| attacks by denying the \verb|noatsecure| permission.
% \begin{minted}{text}
\begin{lstlisting}[style=selinux-cil]
(deny domain tpm_sign_exec_t (file (relabelto)))
(deny domain tpm_sign_file_t (file (relabelfrom)))
(deny domain tpm_sign_t (process (ptrace noatsecure)))
(typeattribute not_tpm_sign)
(typeattributeset not_tpm_sign (not tpm_sign_t))
(deny not_tpm_sign tpm_sign_file_t (file (read)))
(deny not_tpm_sign tpm_sign_t (file (read)))
\end{lstlisting}

We also provide example policy below to explicitly prevent unapproved processes from launching \tpmsign in production systems.
\begin{lstlisting}[style=selinux-cil]
(typeattribute not_tpm_launcher) ; EXAMPLE
(typeattributeset not_tpm_launcher (not tpm_sign_launcher_t))
(deny not_tpm_launcher tpm_sign_t (process (transition)))
\end{lstlisting}

We recommend making SELinux state immutable during runtime with \texttt{setsebool -P secure\_mode\_policyload on}.
To make approved changes, the system can be booted with SELinux in permissive mode with boot parameter \texttt{enforcing=0}.
If this is not feasible, a more complex IMA policy may suffice.

\subsection{IMA Policy}\label{sec:implementation-ima}
While SELinux policy with extensive boot-time measurements may be sufficient to ensure integrity of a system,
we attempted to minimize our reliance on SELinux due to its configuration complexity.
Thus, we turned to IMA (and EVM, \zcref[S]{sec:implementation-evm}) to protect the integrity of the \tpmsign, \wcp, and the rest of our system.

Our IMA policy imitates the default policies \verb|appraise_tcb| and \verb|secure_boot| \cite{IBM:IMA-Policy},
which check the integrity of files owned by root, kernel modules, and other custom IMA policies,
but these rules are not strictly required for protecting \tpmsign and its files.
We then include the following lines at the beginning of our custom policy, because the policy is checked top-to-bottom,
to check the integrity of the \tpmsign binary and files based on our assigned SELinux labels.
To prevent an adversary from labelling malicious binaries as \exect,
which would allow them to access \filet files,
we require our files to be signed with \verb|appraise_type=imasig|.
% \begin{minted}{bash}
\begin{lstlisting}[style=ima-policy]
appraise obj_type=tpm_sign_exec_t appraise_type=imasig
appraise obj_type=tpm_sign_files_t appraise_type=imasig
\end{lstlisting}
% \end{minted}

IMA policy configuration should be flexible enough to ensure integrity on any desired subset of the filesystem.
Thus, IMA may be able to provide defense in cases where SELinux cannot be immutable,
at the risk of increased operational complexity.

\subsection{Memory Protections}
We also want to protect \tpmsign's memory to prevent \wcp from being leaked.
We trust \tpmsign does not have memory safety issues.
We previously solved concerns of direct attacks against \tpmsign with SELinux in \zcref[S]{sec:implementation-selinux}.
\tpmsign further disables coredumps with \verb|prctl|, and disables swapping memory to disk with \verb|mlockall|.
Memory may also be readable from \verb|/proc/kcore|,
\verb|/dev/mem|, and \verb|/dev/kmem|,
which can be protected with SELinux or the \verb|lockdown=confidentiality| boot parameter.
These changes protect \wcp from intended debugging features;
we cannot prevent side-channel attacks.

\subsection{Boot-time and Initramfs Measurements}\label{sec:script}
We add a script to \initramfs that measures the IMA and SELinux policies
as well as the \systemd init binary on the root filesystem during boot.
These measurements indicate whether our framework is properly enabled.
The script then attempts to unlock the LUKS container which contains \wcp.

SELinux policy is uniquely tricky to measure,
as it does not load from a deterministic path, but is instead dependent on the SELinux configuration and newest SELinux policy version the kernel supports.
To solve this easily, we check that only one possible policy file can be loaded, then measure it.
We also measure the SELinux enforcing state according to the configuration file.

The order of the measurements for \systemd, IMA policy, and SELinux policy is inconsequential.
Our script indicates the completion of measurements by writing an extra string to the PCR.
This prevents an adversary from deleting files to skip and then falsify measurements after boot.

It should be possible to load the SELinux and IMA policies from the measurement script,
however, Fedora's built-in SELinux policy is not designed to be loaded before \verb|switch_root| occurs. 
Problematically, IMA policies containing SELinux labels will then fail to load.
Thus, to ensure the policies are loaded correctly after \verb|switch_root|, we measure the \systemd init binary in our \initramfs script in \zcref[S]{sec:script} as \systemd is responsible for loading the policies in this case.
When the measurement script loads the policies, or when the initial IMA policies
measure \systemd (i.e.\ \verb|appraise_tcb|), this is optional.

% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
#!/bin/sh
INIT="$NEWROOT/usr/bin/init"
IMA_POL="$NEWROOT/etc/ima/ima-policy"
tpm2_pcrextend 11:sha256="$(sha256sum "$INIT" | cut -d' ' -f1)"
tpm2_pcrextend 11:sha256="$(sha256sum "$IMA_POL" | cut -d' ' -f1)"

SEL_ENF="$(sed -n 's/^SELINUX=\(.*\)/\1/p' $NEWROOT/etc/selinux/config)"
tpm2_pcrextend 11:sha256="$(echo "$SEL_ENF" | sha256sum | cut -d' ' -f1)"
SEL_TYPE="$(sed -n 's/^SELINUXTYPE=\(.*\)/\1/p' $NEWROOT/etc/selinux/config)"
SEL_DIR="$NEWROOT/etc/selinux/$SEL_TYPE/policy"
count="$(find "$SEL_DIR" -maxdepth 1 -name "policy.*" -print0 | grep -zc .)"
if [ "$count" -eq 1 ]; then
  SEL_POL=$(find "$SEL_DIR" -name 'policy.*')
  tpm2_pcrextend 11:sha256="$(sha256sum "$SEL_POL" | cut -d' ' -f1)"
fi

tpm2_pcrextend 11:sha256="$(echo done | sha256sum | cut -d' ' -f1)"
\end{lstlisting}

Our script then attempts to automatically unlock the LUKS container, nominally \vault, for \wcp.
It first checks if Secure Boot is enabled, and exits early if it is not.
This helps prevent triggering a TPM lockout when maintenance is being done on the system.
Then our script verifies \vault exists, and attempts to unlock and mount it.
\begin{lstlisting}[style=bash]
IMG_PATH="$NEWROOT/home/user/vault.img"
MAPPER_NAME='vault_mapper'
MOUNT_POINT="$NEWROOT/mnt/vault"

mokutil --sb-state | grep -q 'enabled' || exit 0
test -e "$IMG_PATH" || exit 0
systemd-cryptsetup attach "$MAPPER_NAME" "$IMG_PATH" - tpm2-device=auto,keyfile-timeout=0,tries=1,nofail || exit 0
udevadm settle -E "/dev/mapper/$MAPPER_NAME" -t 2 || exit 0
mkdir -p "$MOUNT_POINT"
mount "/dev/mapper/$MAPPER_NAME" "$MOUNT_POINT" 
\end{lstlisting}

Our script stores measurements in PCR 11.
Other relevant PCRs are 4, 7, 8, 9, and 14 \cite{UAPILinuxTPMPCR}.
Only the usage of PCRs 0--7 is defined by TCG standards, and usage of PCRs 8--15 may vary for other boot setups \cite{Trusted-Computing-Group:2023:TCG-PC-Client-PFP}.
The UEFI measures the bootloader into PCR 4.
The GRUB bootloader measures the command line arguments into PCR 8,
and \initramfs and the kernel into PCR 9 \cite{gnu_grub_measured_boot,gentoo_wiki_secure_boot_grub}.
Since our script is installed in \initramfs, its measurements stored in PCR 11 cannot be falsified.
PCR 7 contains the general Secure Boot state, including the keys used to verify bootup.
PCR 14 contains measurements of Machine Owner Keys, including those necessary for loading custom IMA policies.
These measurements are listed in the order they are taken in \zcref[S]{tab:pcr-usage}.
This ordering constructs a chain of trust from the UEFI to the OS,
providing assurance that each individual measurement is trustworthy.

\subsection{TPM Key Setup}
After rebooting the system with our \initramfs changes,
we set up \wcp and \pk.

\subsubsection{Policy Creation}
We create administrator keys for approving wildcard policies with OpenSSL.
For our development system, we have the adminstrator's private key available---in a real setup, it should be held on a different machine.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
openssl genrsa -out policy_key_private.pem 2048
openssl rsa -in policy_key_private.pem -out policy_key.pem -pubout
\end{lstlisting}
% \end{minted}

After creation, the administrator's public key must be loaded in the TPM to create a ``wildcard policy digest''.
Then \pk can require a wildcard policy authenticated by the administrator's public key.
The important file being created here is \verb|wildcard.digest|.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
tpm2_loadexternal -G rsa -C o -u policy_key.pem -c policy_key.ctx -n policy_key.name
tpm2_startauthsession -S session.ctx
tpm2_policyauthorize -S session.ctx -L wildcard.digest -n policy_key.name
tpm2_flushcontext session.ctx
\end{lstlisting}
% \end{minted}

Finally, the wildcard policy that sets the PCR requirements must be created and signed with the private key created earlier.
The file \pcrpolicy is \wcp.
Our commands create the policy on our development machine to easily collect the PCR values,
but the policy can be created on another machine when the approved PCR values are known.
To update \wcp (e.g.\ after system updates), these commands can be repeated.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
tpm2_pcrread -opcrs.sha256 sha256:4,7,8,9,11,14
tpm2_startauthsession -S session.ctx
tpm2_policypcr -S session.ctx -l sha256:4,7,8,9,11,14 -f pcrs.sha256 -L pcr.wcp
tpm2_flushcontext session.ctx
openssl dgst -sha256 -sign policy_key_private.pem -out pcr.sig pcr.wcp
\end{lstlisting}
% \end{minted}

\subsubsection{Key Creation}
First we create a primary key,
then we generate \pk as a child of the primary key.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
tpm2_createprimary -C o -g sha256 -G aes128cfb -c prim.ctx
tpm2_create -u key.pub -r key.priv -g sha256 -G 'rsa2048:rsapss:null' -a 'sign|fixedtpm|fixedparent|sensitivedataorigin' -C prim.ctx -c signing.ctx -L policy/wildcard.digest
tpm2_readpublic -c signing.ctx -o key.pem --format=pem
rm prim.ctx signing.ctx
\end{lstlisting}
% \end{minted}
The important files here are \verb|key.pub| and \verb|key.priv|, the public and private portions of \pk.
We also create a PEM-formatted copy of the public key for sharing.

\subsubsection{Using the Signing Key}
We give the \exect label to \tpmsign, the only program allowed to access \wcp.
It attempts to reload \pk from a session or re-generates the primary key and loads \pk from its file.
After loading, every use of \pk must follow the standard protocol to authorize key usage with \wcp \cite{tpm2-software:2026:policyauthorize}.

\subsection{Using EVM on \wcp}\label{sec:implementation-evm}
After giving \wcp the \filet label,
we apply the portable EVM (and IMA) signature to protect it via \verb|evmctl| \cite{IBM:IMA-utilities-evmctl}.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
evmctl sign --imasig --portable pcr.wcp -k key.priv
\end{lstlisting}
% \end{minted}
This makes it impossible to change the SELinux labels, file owners, or POSIX permissions of \wcp.
In a real environment, the private key should be held on a separate machine
and the signature should be assigned manually via \verb|setfattr|.

We found enabling EVM without HMAC support is not well-supported on Fedora.
As mentioned in \zcref[S]{sec:evm}, this means EVM may replace non-portable signatures with HMACs, allowing metadata to be modified.
In this case, using a \emph{portable} EVM signature on \wcp is absolutely necessary.

\subsection{LUKS Setup}
% TODO : maybe cite something here?
We set up the \vault LUKS container that unlocks with the same PCRs as \wcp.
Then \wcp must be placed in the container, which protects \wcp via encryption during unapproved boots so it cannot be trivially leaked.
Furthermore, it protects \wcp's SELinux label so the label cannot be changed such that our protections would no longer apply.
Steps 3--5 can also be used to update the PCRs that unlock \vault.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
fallocate -l 1GB ~/vault.img
cryptsetup -v luksFormat ~/vault.img
losetup --find --show ~/vault.img
systemd-cryptenroll --wipe-slot=tpm2 --tpm2-device=auto --tpm2-pcrs=4+7+8+9+11+14 /dev/loop0
losetup -d /dev/loop0
cryptsetup -v luksOpen ~/vault.img vault_mapper
mkfs -t ext4 /dev/mapper/vault_mapper
mkdir -p /mnt/vault/
mount /dev/mapper/vault_mapper /mnt/vault
\end{lstlisting}
% \end{minted}

\section{Evaluation}\label{sec:evaluation}
We experimentally demonstrated the security of our key protections.
Our framework should ensure that during an unapproved boot when protections might be disabled,
it is non-trivial, if not impossible, for an adversary to disable our protections during a future approved boot.
Fortunately, it is easy to detect any modifications to components measured into PCRs such as our measurement script and IMA and SELinux policies,
which will always result in an unapproved boot state where \wcp is locked in its LUKS container.
Thus, we only need to be concerned about modifications to the \tpmsign and \wcp files
and any other unintended ways to use \pk.

When our system is booted in an \textbf{unapproved state}:
        \begin{itemize}
          \item The LUKS container will not open without a password, so \wcp is inaccessible.
                Thus, \wcp cannot be relabelled, which would disable enforcement of our protections on approved boots.
          \item If the LUKS container is manually unlocked with a password by an administrator,
          the PCRs will not match \wcp so \pk cannot be used to produce signatures during an unapproved boot.
          This is the only scenario where an adversary could potentially copy \wcp's contents elsewhere or
          disable our SELinux protections by removing the \filet label from \wcp.
        \end{itemize}

When our system is booted in an \textbf{approved state}:
        \begin{itemize}
          \item It is not possible to corrupt our measurement script to lie about measurements taken,
          which could leave the PCRs in an approved state when the system is not.
          \item It is not possible to corrupt our policies in a way that disables our protections.
          \item It is not possible to load custom kernel modules that are not approved by the administrator,
          which could allow access to kernel memory.
          \item It is not possible to leak \wcp through debugging mechanisms.
          \item It is not possible remove the \filet label from \wcp,
          which would disable our SELinux protections.
          \item It is not possible to run unapproved programs with the \exect label,
          which would allow access to \wcp.
          \item Ultimately, it is not possible to access \wcp or use \pk without going through \tpmsign during an approved boot.
        \end{itemize}

While our LUKS container strategy should protect \wcp during unapproved boots,
it is sometimes necessary to manually unlock the LUKS container to approve new PCRs after a system update.
As opposed to using a password to use \pk, which would not verify the boot state,
using \wcp to also check PCRs provides some extra assurance that \pk still cannot be used.
Note that \wcp is not necessarily unprotected during an unapproved boot if something other than our protections has changed a PCR value.
In practice, an administrator should not manually unlock the LUKS container when \wcp would not be protected.

\section{Limitations}\label{sec:limitations}
There are three primary limitations of our work, two being related to the mechanisms we use.
The first is protecting a large subset of a system with IMA can make the system annoying to change or update.
The second is the speed of the TPM.
Even with optimizations, a TPM can only perform about 20 elliptic curve signatures per second, and fewer RSA signatures.
This may be impractically slow, but other hardware security modules or virtual TPMs may improve on this.

The final limitation is if \wcp is leaked to the adversary, then \pk will be persistently available to the adversary until boot PCRs change (e.g.\ via a system update).
To defend against this, we rely on the assumption that IMA/SELinux protections cannot be bypassed under an approved boot.
However, adversaries may be capable of leaking \wcp through rootkits or other exploits.
We can demonstrate a rootkit that corrupts IMA/SELinux behavior,
however, it is prohibitively difficult to install when IMA checks kernel modules.
We addressed trivial exploits (intentional features) that leak arbitrary memory,
but cannot prevent side-channel attacks, other than making
\tpmsign short-lived to limit opportunity for such attacks.
We believe bypassing our framework on an uncorrupted, up-to-date kernel requires a zero-day or otherwise unmitigable exploit.
We describe potential mitigations for this in \zcref[S]{sec:variations}.

\section{Variations and Future Work}\label{sec:variations}
We considered two relatively straightforward modifications to our implementation.
First, using a unified kernel image could reduce boot and measurement complexity.
Second, writing a minimal, bespoke kernel module to protect \wcp would avoid the need for intricate SELinux and IMA policies and the associated administrative overhead.
Each change would reduce the trusted computing base by reducing the amount of configuration that needs to be verified.

We also considered protecting the TPM-encrypted \pk file as a secret instead of \wcp.
We view this as a strictly worse option---%
any sessions with \pk loaded must also be protected, or sessions, and their performance benefits, must be foregone entirely.
This also rules out the option of storing \pk in the TPM's NVRAM.

To partially address our final limitation, concerning authenticity across boot cycles, two other key management strategies are viable.
Policies can require a check to the boot-count, so a new wildcard policy can be approved for every boot.
Alternatively, a new signing key can be created on every boot, but remote parties would need to verify the PCRs to ensure the signing key is properly protected
(and verify it is TPM-resident, if desired).
These strategies limit runtime leaks to a single boot cycle, at the cost of more complexity in policy or key management.

While other hardware security modules with similar functionality to the TPM exist, they are not widely available on consumer hardware.
Trusted execution environments (TEEs) are commonly available on enterprise hardware,
and provide strong memory isolation protections that
may be sufficient to justify storing \wcp or even \pk directly in protected memory.

Although not currently available,
better support for TPM localities or resettable PCRs in Linux could significantly harden our framework.
Either option would remove the secrecy requirement for \wcp,
allowing us to simplify the SELinux policy and remove EVM and LUKS.
If TPM localities were protected by SELinux (or equivalent),
\wcp could require a locality only accessible by \tpmsign under an approved boot.
Alternatively, the OS could control resettable PCR 23 to include details about the program currently accessing the TPM,
with \wcp verifying those details.
Although we must still assume an adversary cannot corrupt the kernel or SELinux at runtime,
\wcp not needing to be secret negates concern about information leaks.
Pursuing one of these options is our ideal way forward.

\section{Related Work}
K\"uhn et al. \cite{Kuhn:Realizing-Property-Based-Attestation:2007} propose a measured boot strategy indicating the existence of good properties rather than strict binary measurement.
Their technique could improve flexibility of our boot measurements.

CoCoTPM \cite{Pecholt:CoCoTPM-Secure-vTPM:2022} and Hybrid TPM \cite{Kim:hTPM-Hybrid-TPM:2019} provide alternative TPM implementations.
CoCoTPM develops secure virtual TPMs, useful for situations where multiple TEEs rely on a TPM.
Hybrid TPM develops a strategy to combine hardware and software TPMs to improve TPM performance, which is a limitation of the TPM in our usecase.

SeCReT \cite{Jang:SeCReT:2015} establishes a secure communication channel between the ARM TrustZone TEE and the rich execution environment.
To protect session keys, they make minor modifications to the Linux kernel to flush keys from memory and protect page accesses.
This strategy is potentially adaptable to improve our memory protection.

In RESEKRA \cite{Gomez:RESEKRA:2022},
a third-party verifies the IMA measurement log/PCR to determine integrity of programs on a system,
signing new wildcard policies when the measurement log updates.
% As verifiers should not share their wildcard policy signing keys, each verifier will require its own corresponding private key on the edge system.
% In addition, the heavy TPM-based attestation process must be performed for every update, sacrificing even more performance than our framework.
However, it does not address our direct concern that a malicious entity,
through exploits undetected by IMA, may access the wildcard policy and use the key.

Wagner et al. \cite{Wagner:Distributed-Usage-Control-SGX:2018} develop a similar technique to ours using the TPM and the Intel Software Guard Extensions (SGX) TEE.
They acknowledge the potential existence of a TPM-only technique using IMA, EVM, and access controls,
but they miss our realizations that wildcard policies can be treated as secrets and LUKS can protect them during unapproved boots,
making our framework stronger than they expected.
Additionally, SGX is now deprecated on consumer processors without a suitable alternative \cite{Intel:Core12th-SGX-Deprecation:2025}.

\section{Conclusion}
Our framework provides authentication for OS entities and TPM keys using SELinux, IMA, EVM, LUKS, and a simple measurement script,
without restricting access to the TPM or using vendor-specific features like TEEs.
Since the TPM does not have an equivalent authentication mechanism built-in,
our framework substantially improves key security, and thus trustworthiness, against root-level adversaries,
unless they can develop a novel kernel exploit or leverage a side-channel attack.
Fundamentally, our framework is modular such that the configuration complexity may be simplified using alternative mechanisms,
and implementing support for underutilized TPM features could further improve the security of keys.
