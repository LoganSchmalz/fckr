\section{Introduction}
Public key cryptography is a ubiquitous component of modern security: most Internet communication is encrypted and authenticated.  % signed ?
However, it relies on the assumption that private keys are kept secret.
In small-scale, personal settings, users can reasonably trust one another to protect their keys.
In contrast, large-scale systems such as web servers handle thousands of requests per second, and the security of their keys depends entirely on machine configuration and automated processes.
In such settings, keys must be both highly available and strongly protected.

A root-level compromise to a system is often catastrophic,
and systems are setup to prevent such events.
Despite best practices, adversaries sometimes gain root capabilities.
This raises a critical question: how can secrets such as private keys be protected even from a root-level attacker, while remaining usable by legitimate services?

The Trusted Platform Module (TPM) 2.0 can store and use keys outside of main memory so they are never exposed,
and supports authorization mechanisms to access them.
However, it lacks integration with OS concepts such as users or programs.
When manual authorization is infeasible,
the TPM has no built-in mechanism that meaningfully \textit{authenticates} OS entities to restrict key usage.
At the same time, the TPM supports measured boot which establishes firmware and software integrity.
Thus, it is possible to ensure proper authentication is enforced at the OS level, before allowing keys to be used.

% TODO : we present a general methodology and demonstrate how to implement it with...?
In this work, we demonstrate how to combine 
Security-Enhanced Linux (SELinux), the Linux Integrity Measurement Architecture (IMA),
the Extended Verification Module (EVM), and Linux Unified Key Setup (LUKS) with
measured boot and TPM policy authorization to enforce fine-grained access control to TPM-resident secrets.
% We describe the required SELinux, IMA, and EVM configurations, boot-time measurements, LUKS setup, and TPM configuration necessary to achieve this.
We discuss variations and limitations of our setup,
and note that deeper kernel support leveraging existing TPM features could enable stronger future designs.

\subsection{Trusted Platform Module (TPM) 2.0}
The TPM 2.0 is widely available on modern x86 systems,
either as firmware TPMs (fTPMs) integrated in AMD and Intel platforms or as discrete hardware modules,
making it a practical target for our work.
The behavior of the TPM is standardized by the Trusted Computing Group (TCG) \cite{Trusted-Computing-Group:2025:TPM-2.0-Library}.
The TPM serves as a root of trust of storage and root of trust of reporting.
When used with Secure Boot-enabled firmware, it is used to store boot measurements.

\subsubsection{Recording Boot Measurements}
To store boot measurements, the TPM utilizes Platform Configuration Registers (PCRs) \cite{Trusted-Computing-Group:2025:TCG-TPM-2.0-Part1-Architecture}.
PCRs store hashes of values through hash extension operations.
If a PCR has a current hash value $x$ and it is being extended with a new hash $y$,
the PCR's value becomes $x_{new} = \hash{\mathop{concat}(x,y)}$.
By correlating each hash extension to its source,
successfully reconstructing a PCR value demonstrates integrity of the recorded measurements.
For cryptographically secure hash functions like SHA-256,
finding $y$ such that $x_{new}$ lies about previous measurements is infeasible.

PCR extension is the primary mechanism behind Secure Boot and other measured/trusted boot techniques.
During boot, the Unified Extensible Firmware Interface (UEFI) measures the firmware and the bootloader,
and the bootloader measures the kernel and parts of the OS.
Because users cannot interfere with early boot,
the measurements reliably indicate firmware and software integrity.
The measurements remain trustworthy at runtime as they are stored in PCRs 0--15, which are non-resettable and persist until reboot \cite{Trusted-Computing-Group:2023:TCG-PC-Client-PFP}.

\subsubsection{Loading and Storing Secrets}
% TODO : cite here
Since the TPM has limited storage, secrets are often unloaded and reloaded,
and they are stored as a hierarchy.
Primary keys are deterministically generated and act as root parents in the hierarchy.
Their children are regular, randomly generated keys and other secrets.
Parent keys encrypt their children for external storage,
so they must be regenerated (primary keys) or reloaded (regular keys) to decrypt their children.

To improve performance of reloading keys, the TPM can reload a session where the key was already loaded if the system has not rebooted.
Keys can also be persistently stored in the TPM's non-volatile memory (NVRAM) for a further minor improvement.
Loading a key in the TPM merely caches it; authorization checks are still enforced upon use.

\subsubsection{Protecting Keys and Other Secrets}\label{sec:key-protections}
TPM-resident private keys are only usable within the TPM itself and are never available on the OS.
Other secrets have the same protections available, but get released to the OS when those protections are satisfied.
We focus on TPM-resident keys, without loss of generality.

TPM-resident keys support conditional usage,
optionally requiring a password, HMAC, particular PCR values,
or a more general ``policy'' to be satisfied.
Password- or HMAC-based authorization requires a secret to be known at use time,
but such secrets have no inherent protection against disclosure.
PCRs effectively limit key usage to an administrator-approved boot configuration.
Keys are tied to their authorization mechanism.
For password-authorized keys, it is possible to change the password.
For simple PCR-based authorization,
the expected values cannot be changed,
which is problematic for system updates \cite{Arthur:2015:A-Practical-Guide-to-TPM-2.0-Certifying-Keys,Trusted-Computing-Group:2025:TCG-TPM-2.0-Part1-Architecture}.

Fortunately, policy-based authorization is more advanced.
Flexible policies can be made via ``wildcard policies'' or ``authorized policies'' \cite{Arthur:2015:A-Practical-Guide-to-TPM-2.0-Certifying-Keys}.
Wildcard policies must be signed by an administrator with a private key,
and the corresponding public key is included in the TPM-resident key's creation details.
Policies can also include PCR checks.
Thus, wildcard policies provide a way to change PCR requirements
so keys remain usable even when boot measurements change.
Wildcard policies may also be leveraged to allow multiple policies to authorize usage of a single key,
or to utilize the same policy for different keys, including keys on different devices.

\subsection{Security-Enhanced Linux (SELinux)}
SELinux \cite{Loscocco:2001:SELinux} provides mandatory access controls on Linux and is integrated into the Linux kernel. 
Labels added to users, files, and processes determine their permissions according to the SELinux policy installed on the system.
These permissions can control actions such as reading and writing to certain files (including append-only writes),
executing programs, inter-process communi\-cation, and opening specific network ports.

\subsection{Integrity Measurement Architecture (IMA)}
IMA \cite{Sailer:2004:Integrity-Measurement-Architecture,IBM:IMA-Documentation} measures files and prevents them from being opened or executed if they are changed from an approved state.
Like SELinux, IMA is widely available.
Integrity hashes and signatures for files are stored as extended filesystem attributes.
When IMA is enabled in \fix mode, files matched by the IMA policy have their hashes updated.
In \appraise mode, IMA prevents protected files from being opened if their contents do not match the associated hash.
IMA \fix mode can only be enabled when Secure Boot is disabled in the UEFI,
otherwise \appraise mode is enabled.
IMA policy can include various specifications such as file owners, the active user, and whether files are being opened for reading or execution.

Signatures can also be used to detect file corruption.
Allowing hashes to be overwritten in \appraise mode is unsafe, as an adversary may know the expected value.
To switch to \fix mode from \appraise mode to update hashes, a reboot is required.
In contrast, signatures can be updated safely in \appraise mode because the administrator's signing key remains protected.
This makes it easier to install and update programs and files.

By default, some distributions require an administrator to sign any custom IMA policies and public keys.
This involves adding a key certificate to the UEFI Machine Owner Keys, then signing IMA policies and X.509 keys on the filesystem with the appropriate private key.
The first custom IMA policy that is loaded overwrites any default policies.
Afterwards, the custom IMA policy cannot be overwritten, only appended to.

\subsection{Extended Verification Module (EVM)}
\label{sec:evm}
% TODO: what to cite here? https://lwn.net/Articles/393673/ ima-docs?
EVM is a similar integrity mechanism to IMA, but protects file meta\-data, such as the permissions and SELinux label.
This prevents adversaries from simply adjusting file metadata to bypass SELinux or IMA protections.
EVM can use HMAC to detect any offline filesystem modifications, or use signatures to detect runtime modifications;
both can be enabled at once.
When a file is appraised by IMA, its EVM HMAC/signature is verified.

``Portable'' EVM signatures exclude system-specific metadata and
actively prevent modifying metadata: although IMA prevents \emph{the operation} from occuring \emph{if the file is corrupt},
it does not inherently prevent initial corruption from occuring;
when using EVM portable signatures, any modification to the file's \emph{metadata} will be prevented.
When both HMACs and signatures are enabled, EVM may replace non-portable signatures with HMACs upon metadata changes.

\subsection{Linux Unified Key Setup (LUKS) Containers}
LUKS \cite{Fruhwirth:LUKS:2005} creates encrypted filesystem images, protected with a password or TPM PCRs.
Thus, LUKS can protect files when the system boots into an unapproved state.
Whether a LUKS-based or a manual decryption process is used, a key must be kept in memory or otherwise easily available.
Since LUKS containers trivially retain file metadata and are managed by the kernel,
we favor the widespread, easy-to-use LUKS mechanisms over a manual decryption process.

\section{Motivation}
We assume an adversary has root privileges on a general purpose Linux system.
Without our framework, the adversary can read or write any file,
install kernel modules, dump program memory, modify the boot setup, and reboot the system;
thus, they can access the TPM and any files necessary for a particular request.
We assume the adversary has no physical access to the machine, so the hardware and firmware are trustworthy.
We want to avoid artificially restricting access to the TPM or other system components.

Notice we did not point out a mechanism for the TPM to authenticate OS entities in \zcref[S]{sec:key-protections}.
TPM policy does not have any straightforward implementation for this.
The TPM has a feature called ``localities'' which is intended to be used to indicate the origin of a command (hardware, kernel, user-level software) \cite{Trusted-Computing-Group:2023:TCG-PC-Client-PFP,Arthur:2015:A-Practical-Guide-to-TPM-2.0-Certifying-Keys}.
In theory, localities could be tied to entities to restrict access to a key.
However, the TPM trusts the OS to control localities, and there is no existing mechanism, SELinux included, designed to accomplish this.
Alternatively, one of the resettable PCRs, controlled by the OS, could indicate the entity currently accessing the key.
This is also not currently supported.
In practice, any entity allowed to access the TPM (e.g. the root adversary) can make requests in place of other entities without distinction, unless manual input is required.
Any viable automatic option relies on the OS to perform authentication.

We provide a solution to authenticate TPM key usage leveraging existing security concepts.
If a secret is required to use our key and the secret is well-protected, it authenticates entities by distinguishing those that can and cannot access the secret.
If integrity of the entities is ensured, only the intended, legitimate entities can access the secret and use the key.
This suggests the secret must only be available (LUKS) when access controls (SELinux) and integrity (IMA, EVM) are being enforced (measured boot).
Importantly, these protections must apply to root-level users,
and must prevent alternative means of leaking the secret like installing kernel modules or inspecting program memory.
This concept properly authenticates entities accessing the key, working around the limited existing integration between the TPM and the OS.
Any of our chosen mechanisms (including the TPM) could be swapped for an alternative design with the same principles.

\section{Framework}
In our framework, access to a TPM signing key \pk is restricted to a single program \tpmsign.
SELinux enforces rules on \tpmsign and a wildcard policy \wcp.
IMA protects the integrity of \tpmsign, as well as any binaries permitted to launch it.
EVM and SELinux protect the SELinux labels on \tpmsign and \wcp.
%These protections are strongest when the SELinux policy is designated immutable,
%preventing runtime policy modification by an adversary.

The wildcard policy \wcp is the secret used to access \pk.
\wcp is stored within a LUKS container that unlocks during boot when our framework is enabled.
When the system is booted into an unapproved state, \wcp is encrypted inside the LUKS container and cannot be accessed.
Additionally, \wcp itself requires the PCRs to indicate the system booted with our framework.
If the container is unlocked manually while \wcp is unprotected, the PCRs will indicate such, so \pk remains unusable.
Thus, for an adversary to access \wcp, they must exploit the running system to bypass the IMA or SELinux policies,
or otherwise find an exploit to read \wcp from memory.
The relationships of mechanisms in our framework are visually demonstrated in \zcref[S]{fig:framework}.

We attempted to minimize our reliance on SELinux since policy development can get quite complex,
especially as support for deny rules was conveniently finalized during our research.
Further, for our research outside this particular work, we must protect other infrastructure which interacts with
the \tpmsign binary.
Thus, we turned to IMA and EVM to protect the integrity of the \tpmsign binary and the SELinux labels.
While SELinux with more boot-time measurements may be sufficient,
we believe IMA is easy to grasp and adds an additional, more dynamic layer of protection.

\subsection{IMA Policy}
IMA protects the integrity of \tpmsign binary and relevant files.
Our policy imitates the default \verb|tcb| and \verb|appraise_tcb| policies \cite{IBM:IMA-Policy},
then we include the following lines at the beginning:
% \begin{minted}{bash}
\begin{lstlisting}[style=ima-policy]
appraise obj_type=tpm_sign_exec_t appraise_type=imasig
appraise obj_type=tpm_sign_files_t appraise_type=imasig
\end{lstlisting}
% \end{minted}
We require signatures for these files with \verb|appraise_type=imasig|,
which is stronger than only requiring hashes.
The first rule effectively prevents an adversary from relabeling a malicious binary to \exect,
since all files with that label require an administrator's signature.
% ! TODO : consider removing second rule?

The flexibility of IMA policy configuration should make it possible to ensure integrity on any desired subset of the filesystem.
Complex policies may be able to provide defense on systems where the SELinux state cannot be immutable,
at the risk of increased operational complexity.
At the least, we recommend including the rules of the available default IMA policies. % TODO : cite default rules page

\subsection{Boot-time and Initramfs Measurements}
We add a script to \initramfs that measures the IMA and SELinux policies
as well as the \systemd init binary on the root filesystem during boot.
The script then attempts to unlock the LUKS container which contains \wcp.

% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
#!/bin/sh
INIT="$NEWROOT/usr/bin/init"
SELINUX_POLICY=$(find "$NEWROOT/etc/selinux/targeted/policy" -maxdepth 1 -name 'policy.*' | sort -V | tail -n 1)
IMA_POLICY="$NEWROOT/etc/ima/ima-policy"

INIT_SHA=$(sha256sum "$INIT" | cut -d' ' -f1)
tpm2_pcrextend 11:sha256="$INIT_SHA"
# ... similar for SELinux, IMA
HASH=$(printf 'done' | sha256sum | cut -d' ' -f1)
tpm2_pcrextend 11:sha256="$HASH"

IMG_PATH="$NEWROOT/home/user/vault.img"
MAPPER_NAME='vault_mapper'
MOUNT_POINT="$NEWROOT/mnt/vault"

mokutil --sb-state | grep -q 'enabled' || exit 0
test -e "$IMG_PATH" || exit 0
systemd-cryptsetup attach "$MAPPER_NAME" "$IMG_PATH" - tpm2-device=auto,keyfile-timeout=0,tries=1,nofail || exit 0
udevadm settle -E "/dev/mapper/$MAPPER_NAME" -t 2 || exit 0

mkdir -p "$MOUNT_POINT"
mount "/dev/mapper/$MAPPER_NAME" "$MOUNT_POINT" 
\end{lstlisting}

Since this script is installed in \initramfs,
removal of the script or other unapproved modifications of \initramfs will be reflected in a PCR.
Thus, it is not possible for an adversary to falsify the measurements made by this script. % ! (or its absence).

\subsection{TPM Key Setup}
After rebooting the system with our \initramfs changes,
we set up \wcp and \pk.

\subsubsection{Policy Creation}
For our development system, we have the adminstrator's private key available---in a real setup, it should be held on a different machine.
We create the administrator keys with OpenSSL.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
openssl genrsa -out policy_key_private.pem 2048
openssl rsa -in policy_key_private.pem -out policy_key.pem -pubout
\end{lstlisting}
% \end{minted}

The administrator's public key must be loaded in the TPM to create a ``wildcard policy digest''.
Then \pk can require a wildcard policy authenticated by the administrator's public key.
The important file being created here is \verb|wildcard.digest|.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
tpm2_loadexternal -G rsa -C o -u policy_key.pem -c policy_key.ctx -n policy_key.name
tpm2_startauthsession -S session.ctx
tpm2_policyauthorize -S session.ctx -L wildcard.digest -n policy_key.name
tpm2_flushcontext session.ctx
\end{lstlisting}
% \end{minted}

The wildcard policy that sets the PCR requirements must be created and signed with the private key created earlier.
The file \pcrpolicy is \wcp.
Our commands create the policy on our development machine to easily collect the PCR values,
but the policy can be created on another machine when the approved PCR values are known.
To update \wcp (e.g. after system updates), these commands can be repeated.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
tpm2_pcrread -opcrs.sha256 sha256:4,7,8,9,11,14
tpm2_startauthsession -S session.ctx
tpm2_policypcr -S session.ctx -l sha256:4,7,8,9,11,14 -f pcrs.sha256 -L pcr.wcp
tpm2_flushcontext session.ctx
openssl dgst -sha256 -sign policy_key_private.pem -out pcr.sig pcr.wcp
\end{lstlisting}
% \end{minted}

\subsubsection{Key Creation}
First we create a primary key,
then we generate \pk as a child of the primary key.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
tpm2_createprimary -C o -g sha256 -G aes128cfb -c prim.ctx
tpm2_create -u key.pub -r key.priv -g sha256 -G 'rsa2048:rsapss:null' -a 'sign|fixedtpm|fixedparent|sensitivedataorigin' -C prim.ctx -c signing.ctx -L policy/wildcard.digest
tpm2_readpublic -c signing.ctx -o key.pem --format=pem
rm prim.ctx signing.ctx
\end{lstlisting}
% \end{minted}
The important files here are \verb|key.pub| and \verb|key.priv|, the public and private portions of \pk.
We also create a PEM-formatted copy of the public key for sharing.

\subsubsection{Using the Signing Key}
We make \tpmsign the only program allowed to access
\wcp (\pcrpolicy) by giving it the \exect label.
It attempts to reload \pk from a session;
otherwise, it re-generates the primary key and loads \pk from its file.
After loading, every use of \pk must follow the standard protocol to authorize key usage with the wildcard policy \cite{tpm2-software:2026:policyauthorize}.

\subsection{Using EVM on \wcp}
After giving \wcp the \filet label,
we apply the portable EVM (and IMA) signature to protect it via \verb|evmctl| \cite{IBM:IMA-utilities-evmctl}.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
evmctl sign --imasig --portable pcr.wcp -k key.priv
\end{lstlisting}
% \end{minted}
This makes it impossible to change the SELinux labels, file owners, or UNIX permissions of \wcp.
In a real environment, the private key should be held on a separate machine
and the signatures should be assigned manually via \verb|setfattr|.

\subsection{LUKS Container}
% TODO : maybe cite something here?
We set up a LUKS container, nominally \vault, that unlocks with the same PCRs as \wcp.
Then \wcp must be placed in the \vault.
The same \texttt{losetup} and \texttt{systemd-cryptenroll} commands update the PCRs for the container.
% \begin{minted}{bash}
\begin{lstlisting}[style=bash]
fallocate -l 1GB ~/vault.img

losetup -d /dev/loop0
cryptsetup -v luksOpen ~/vault.img vault_mapper
mkfs -t ext4 /dev/mapper/vault_mapper
mkdir -p /mnt/vault/
mount /dev/mapper/vault_mapper /mnt/vault
\end{lstlisting}
% \end{minted}

\subsection{Deployment Considerations}\label{sec:implementation-details}
Our development system uses Fedora 42 with an out-of-the-box kernel.
This influenced some of our design decisions which may require modification for other distributions.

Many Linux distributions use shim, a simple Microsoft-signed bootloader that facilitates Secure Boot support \cite{rhboot:2026:shim}.
Shim contains its own certificate database to check signatures, so
the UEFI measures shim, and shim measures the real bootloader (e.g. GRUB).
This does not affect the security of our framework, as the chain of trust remains intact.
Alternatively, administrators may manually sign the bootloader or use a unified kernel image (UKI) \cite{gentoo_wiki_secure_boot,gentoo_wiki_secure_boot_grub}.

While PCR 11 is unused and available on Fedora when using the GRUB bootloader, only the usage of PCRs 0--7 is defined by TCG standards.
Usage of PCRs 8--15 may vary for other boot setups \cite{Trusted-Computing-Group:2023:TCG-PC-Client-PFP,UAPILinuxTPMPCR}.

We found that attempting to run EVM without HMAC support prevents Fedora from booting due to its reliance on dynamically-modified files.
As mentioned in \zcref[S]{sec:evm}, this means EVM may replace non-portable signatures with HMACs, allowing metadata to be modified.
In this case, \textbf{portable} signatures are absolutely necessary.

Finally, on some distributions it should be possible to load the SELinux and IMA policies from the measurement script.
However, Fedora's built-in SELinux policy is not designed to support loading before \verb|switch_root|,
and IMA policies containing SELinux labels will fail to load if the SELinux policy is not loaded beforehand.
To ensure the policies are loaded, we measure the \systemd init binary as it is responsible for loading the policies after \verb|switch_root|.
When the measurement script loads the policies, or when the initial IMA policies
measure \systemd (i.e. \verb|appraise_tcb|), this is optional.


\section{Evaluation and Limitations}
We experimentally demonstrated the following features of this setup:
\begin{itemize}
  \item Changing the \initramfs, kernel, IMA policy, SELinux policy, and other items measured into PCRs results in an unapproved boot state.
  \item When the system is booted in an unapproved state:
        \begin{itemize}
          \item The LUKS container will not open without a password, and \wcp is inaccessible.
                Thus, \wcp cannot be relabelled.
          \item If the container is manually unlocked so \wcp is available, the PCRs will not match \wcp and \pk will not be usable.
          \item It is not possible to corrupt the IMA policy, because custom IMA policies require a signature.
        \end{itemize}
  \item When the system is booted in an approved state (IMA and SELinux working as intended), it is not possible to:
        \begin{itemize}
          \item Give unapproved programs the \exect label
          \item Corrupt the \tpmsign binary to leak information
          \item Remove the \filet label from \wcp
          \item Corrupt the IMA or SELinux policies
          \item Access \wcp or use \pk without going through \tpmsign
        \end{itemize}
\end{itemize}

While the LUKS container should protect \wcp (or a password) sufficiently,
it is sometimes necessary to manually unlock the LUKS container when the system is in an unapproved state to accomodate a system update.
Since it is still beneficial to know \pk cannot be used at that moment,
\wcp also checking PCRs provides some extra assurance that if \pk is used, then the system booted in an approved state.
Note that \wcp is not necessarily unprotected during an unapproved boot, as the IMA and SELinux policies may still be approved but another part of the system may have changed.
To exploit a lapse in protection caused by a manual unlock, an adversary must remember the contents of \wcp until the system is rebooted to an approved state.

There are three primary limitations of this secret protection scheme.
The first is protecting a large subset of a system with IMA can make the system annoying to change or update.
This is an acknowledged problem, but an unfortunate reality of using IMA.

The second limitation is assuming IMA and SELinux protections cannot be bypassed under an approved boot.
Against an adversary capable of installing a rootkit or otherwise corrupting the kernel at runtime,
the contents of \wcp can be leaked such that it, and thus \pk, will be persistently available to the adversary until boot PCRs change (e.g. via a system update),
regardless of the state of SELinux and IMA.
We can demonstrate a rootkit that corrupts IMA/SELinux behavior,
however, on a system where IMA checks kernel modules, it is prohibitively difficult to install.
In addition, our framework does not independently defend against exploits that leak arbitrary memory, like side-channel attacks,
but \tpmsign is short-lived to limit opportunity for such attacks.
Thus, we believe bypassing our framework on an otherwise uncorrupted, up-to-date kernel has equivalent difficulty to finding at least one zero-day exploit.

The final limitation is the speed of the TPM.
Even with optimizations, a non-virtualized TPM typically takes a few hundred milliseconds to complete the sequence of operations required for signing.
This is problematic in environments requiring high-frequency operations.
Alternative hardware security modules or virtualized TPMs may improve on this.

\section{Variations and Future Work}
% The design of our framework allows for several adjustments to address specific concerns.

We considered two relatively straightforward modifications to our implementation.
First, using a unified kernel image could reduce boot and measurement complexity.
Second, writing a minimal, bespoke kernel module to protect \wcp would avoid the need for intricate SELinux and IMA policies and the associated administrative overhead.
Each change would reduce the trusted computing base by reducing the amount of configuration that needs to be verified.

We also considered protecting the TPM-encrypted \pk file as a secret instead of \wcp.
We view this as a strictly worse option---%
any sessions with \pk loaded must also be protected, or sessions, and their performance benefits, must be foregone entirely.
This also rules out the option of storing \pk in the TPM's NVRAM.

If there is serious concern about authenticity across boot cycles, two other key management strategies are feasible.
Policies can require a check to the boot-count, so \wcp could be reprovisioned on every boot.
Alternatively, \pk could be reprovisioned on every boot, but remote parties would need to verify the PCRs to ensure \pk is properly protected
(and TPM-resident, if that is still desired).
These options limit runtime leaks to a single boot cycle, at the cost of increased complexity in policy or key management respectively.

While other hardware security modules with similar functionality to the TPM exist, they are not widely available.
Trusted execution environments (TEEs) are, however, commonly available on enterprise hardware,
and provide strong memory isolation protections.
TEEs may be sufficient to justify storing \wcp or \pk directly in protected memory,
potentially improving performance.

TPM localities or resettable PCRs could be leveraged to significantly harden our framework.
Either option would remove the secrecy requirement for \wcp,
simplifying the SELinux policy and removing the need for EVM and LUKS.
If TPM localities were protected by SELinux (or equivalent),
\wcp could require a locality only accessible by \tpmsign under an approved boot.
Alternatively, the kernel could control resettable PCR 16, and include details about the program currently accessing the TPM in PCR 16.
Then the PCR 16 value for \tpmsign could be included in \wcp.
Unfortunately, adversaries capable of corrupting the kernel/SELinux at runtime could still break either strategy.
This is our ideal way forward.

\section{Related Work}
Haas and Pirker \cite{Haas:Linux-Boot-Integrity:2024} provide a broader overview of the Linux measured boot process.
Their work is supplemental to our brief background, explaining how various Linux boot setups interact with Secure Boot,
as well as referencing useful tools.

K\"uhn et al. \cite{Kuhn:Realizing-Property-Based-Attestation:2007} propose a measured boot strategy indicating the existence of good properties rather than strict binary measurement.
Their technique could improve flexibility of our boot measurements.

CoCoTPM \cite{Pecholt:CoCoTPM-Secure-vTPM:2022} develops secure virtualized TPMs.
For setups utilizing TEEs that rely on TPMs, CoCoTPM provides a flexible and robust solution.

Hybrid TPMs \cite{Kim:hTPM-Hybrid-TPM:2019} improve performance over discrete/firmware TPMs,
which would fix a limitation of our framework.

SeCReT \cite{Jang:SeCReT:2015} establishes a secure communication channel between the ARM TrustZone TEE and the rich execution environment.
To protect session keys, they make minor modifications to the Linux kernel to flush keys from memory and protect page accesses.
This strategy is potentially adaptable to our framework.

In RESEKRA \cite{Gomez:RESEKRA:2022}
a third-party verifies the IMA measurement log and corresponding PCR to determine runtime integrity of programs on a system,
and signs new wildcard policies when the measurement log updates.
% As verifiers should not share their wildcard policy signing keys, each verifier will require its own corresponding private key on the edge system.
% In addition, the heavy TPM-based attestation process must be performed for every update, sacrificing even more performance than our framework.
While RESEKRA presents a valuable technique, it does not address our direct concern that a malicious entity,
through exploits undetected by IMA, may use the wildcard policy to sign falsified data.

\section{Details of SELinux Policy}\label{appendix:selinux}
There are multiple variants on the SELinux policy setup to accomplish the same protections as in the main portion of the text.
Here, we discuss some of the nuances for various choices.
Notably, in almost all of the basic SELinux policy tutorials, these details are not included.
This is largely because most SELinux policies for applications do not need to be as strict as ours, and instead simply need to follow a relatively standard template.

The first important detail to note is the existence of the SELinux Reference Policy. % TODO : cite
The Reference Policy serves as the basis for most out-of-the-box SELinux policies and includes some useful macros which are widely used.
One particular macro is \verb|files_type(x)|, which nominally associates the type \verb|x| with the type \verb|file_type|.
The \verb|file_type| type includes permissions allowing unconfined contexts to access and manipulate files with label \verb|x|.
This is entirely reasonable in most situations where an administrator should be easily allowed to manipulate files.
However, it is not suitable for developing protections such as ours,
as unconfined contexts are given the following permissions:
\begin{lstlisting}[style=selinux-policy]
allow files_unconfined_type file_type:file { exec_file_perms manage_file_perms mounton quotaon relabel_file_perms watch watch_mount watch_reads watch_sb watch_with_perm };
\end{lstlisting}
% TODO: cite https://github.com/SELinuxProject/refpolicy/blob/2cba8023863718709d0349faf62a9f4da2248a3f/policy/modules/kernel/files.te#L230,
% TODO: cite https://github.com/SELinuxProject/refpolicy/blob/2cba8023863718709d0349faf62a9f4da2248a3f/policy/support/obj_perm_sets.spt#L158
We found ourselves concerned with
\verb|exec_file_perms|, which is a macro that includes \verb|open| and \verb|read|.
and with \verb|relabel_file_perms|, which includes \verb|relabelfrom| and \verb|relabelto|.
In Fedora's \verb|targeted| policy specifically, unconfined contexts are also able to transition to any domain type and inspect the memory of any program. % TODO: improve
By parsing through these definitions in the Reference Policy, it is possible to recreate them while excluding undesirable permissions.

The second important detail to note is the implementation of deny rules in the SELinux Common Intermediate Language.
Deny rules are capable of overriding existing policy, however they are not available in the high-level SELinux syntax.
Since deny rules are very explicit, they can compensate for pre-existing SELinux policy modules which leave unintended security holes.
Thus, they are a suitable alternative to the above process of manually recreating simple things which could be based on existing macros.
For example, to recreate our policy while making denials explicit:
\begin{lstlisting}[style=selinux-cil]
(deny domain tpm_sign_exec_t (file (relabelto)))
(deny domain tpm_sign_file_t (file (relabelfrom)))

(typeattribute tpm_sign_file_deny)
(typeattributeset tpm_sign_file_deny (not tpm_sign_t))
(deny tpm_sign_file_deny tpm_sign_file_t (file (read)))
\end{lstlisting}
The \verb|typeattribute|/\verb|typeattributeset| declaration is necessary to determine which types should be denied in the last deny rule
via logical expressions like \verb|not|, \verb|and|, \verb|or|, etc.
This is because, in CIL policy, rules (\verb|allow|, \verb|deny|, etc.) must take an identifier in the first and second arguments.
In our policy, this allows us to deny reads from every domain except \domaint.

\section{Setting up IMA}
Setting up IMA is a complicated process on Fedora.
To ensure trustworthiness of the installed IMA policy and signatures, there are some setup steps that go beyond ``put a file in a place''.
We recommend at least reading the sections starting from ``IMA CA Key and Certificate'' to ``Sign and Install a Custom Policy'' at \cite{IBM:IMA-utilities-evmctl} to gather the basics.

On Fedora, dracut is the tool used to build \initramfs.
Dracut has a standard module for loading IMA/EVM keys and enabling IMA/EVM during boot named \verb|integrity|.
It is necessary to remove the \verb|ima-policy-load.sh| script from this module, since Fedora's SELinux policy is not designed to load during boot and our IMA policy relies on the SELinux policy.
The \verb|integrity| module loads IMA keys from \verb|/etc/keys/ima/| as described in \cite{IBM:IMA-utilities-evmctl}.
Before that, it loads the EVM keys from \verb|/etc/keys/evm/|;
thus, another key with a similar setup (or the same key as generated for IMA in the ``IMA Signing Key and Certificate'' section) must also be placed there to serve as the EVM signature verification key.

As discussed briefly in \zcref[S]{sec:implementation-details}, it seems best on Fedora to enable EVM HMACs as well.
This requires creating a kernel masterkey (we make a trusted masterkey since we are using a TPM) and making a child encrypted EVM HMAC key.
\begin{lstlisting}[style=bash]
tpm2_createprimary --key-algorithm=rsa2048 --key-context=key.ctxt
tpm2_evictcontrol -c key.ctxt 0x81000001
KMK=$(keyctl add trusted kmk-trusted "new 32 keyhandle=0x81000001" @u)
keyctl pipe $KMK > /etc/keys/kmk-trusted.blob
EVM=$(keyctl add encrypted evm-key "new trusted:kmk-trusted 64" @u)
keyctl pipe $EVM > /etc/keys/evm-trusted.blob
\end{lstlisting}
